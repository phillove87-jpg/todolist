# TodoList 앱 PRD

## 1. 개요

### 1.1 목적

`TodoList`는 개인 사용자가 해야 할 일을 간단하게 등록하고, 완료 처리하고, 삭제할 수 있는 웹 기반 할 일 관리 앱이다. 첫 번째 목표는 복잡한 협업 기능 없이도 개인이 자신의 할 일을 빠르게 정리하고 관리할 수 있는 최소 기능 제품(MVP)을 만드는 것이다.

### 1.2 배경

기존 할 일 앱은 기능이 많아질수록 진입 장벽이 높아지고, 단순히 오늘 해야 할 일을 기록하고 관리하려는 사용자에게는 오히려 부담이 될 수 있다. 이 프로젝트는 1인 개발을 전제로, 핵심 사용 흐름에만 집중한 가볍고 명확한 Todo 경험을 제공하는 데 목적이 있다.

### 1.3 제품 목표

- 사용자가 10초 이내에 새 할 일을 등록할 수 있어야 한다.
- 사용자가 자신의 전체 할 일 목록과 완료 상태를 한 화면에서 확인할 수 있어야 한다.
- 사용자가 로그인 후 자신의 데이터만 안전하게 조회하고 관리할 수 있어야 한다.

## 2. 타겟 사용자

### 2.1 주요 사용자

- 개인 할 일 관리를 원하는 일반 사용자
- 복잡한 프로젝트 관리 도구보다 간단한 체크리스트 성격의 앱을 선호하는 사용자
- 모바일과 데스크톱 브라우저 모두에서 빠르게 접근하고 싶은 사용자

### 2.2 사용자 특성

- 팀 협업보다 개인 생산성 관리에 관심이 높다.
- 할 일을 짧은 텍스트 중심으로 기록한다.
- 가입과 로그인 이후 바로 기능을 사용할 수 있는 단순한 UX를 기대한다.

## 3. 사용자 스토리

### 3.1 회원가입

사용자로서, 나만의 할 일 데이터를 저장하기 위해 회원가입하고 싶다.

Acceptance Criteria:
- 사용자는 이메일과 비밀번호를 입력해 회원가입할 수 있다.
- 이미 가입된 이메일로는 중복 회원가입이 불가능하다.
- 회원가입 성공 후 로그인 가능한 상태가 되어야 한다.

### 3.2 로그인

사용자로서, 내가 저장한 할 일 목록에 접근하기 위해 로그인하고 싶다.

Acceptance Criteria:
- 사용자는 등록된 이메일과 비밀번호로 로그인할 수 있다.
- 로그인 성공 시 인증 토큰이 발급되어야 한다.
- 인증되지 않은 사용자는 할 일 목록 API에 접근할 수 없어야 한다.

### 3.3 할 일 추가

사용자로서, 해야 할 일을 잊지 않기 위해 새 할 일을 추가하고 싶다.

Acceptance Criteria:
- 사용자는 할 일 제목을 입력해 새 항목을 생성할 수 있다.
- 제목이 비어 있으면 생성할 수 없어야 한다.
- 생성된 할 일은 목록에 즉시 반영되어야 한다.

### 3.4 할 일 목록 조회

사용자로서, 현재 내가 해야 할 일과 완료한 일을 한눈에 확인하고 싶다.

Acceptance Criteria:
- 사용자는 로그인 후 자신의 할 일 목록만 조회할 수 있다.
- 목록에는 최소한 할 일 제목, 완료 여부, 생성일이 표시되어야 한다.
- 데이터가 없을 경우 빈 상태 메시지가 보여야 한다.

### 3.5 할 일 완료 처리

사용자로서, 끝낸 일을 구분하기 위해 완료 상태를 체크하고 싶다.

Acceptance Criteria:
- 사용자는 개별 할 일을 완료 상태로 변경할 수 있다.
- 완료된 할 일은 시각적으로 구분되어야 한다.
- 사용자는 완료 상태를 다시 미완료로 되돌릴 수 있어야 한다.

### 3.6 할 일 삭제

사용자로서, 더 이상 필요 없는 할 일을 목록에서 제거하고 싶다.

Acceptance Criteria:
- 사용자는 자신의 할 일을 삭제할 수 있다.
- 삭제된 할 일은 목록에서 즉시 사라져야 한다.
- 다른 사용자의 할 일은 삭제할 수 없어야 한다.

## 5. 비기능 요구사항

### 5.1 반응형 UI

- 앱은 모바일 `360px` 이상 화면에서 정상 동작해야 한다.
- 태블릿과 데스크톱 화면에서도 레이아웃이 깨지지 않아야 한다.
- TailwindCSS의 `sm`, `md`, `lg` 브레이크포인트를 기준으로 레이아웃이 전환되어야 한다.
- 모바일에서는 단일 컬럼 중심, 데스크톱에서는 여백과 가독성을 강화한 레이아웃을 제공해야 한다.

### 5.2 성능

- 첫 화면은 일반적인 네트워크 환경에서 빠르게 로드되어야 한다.
- 할 일 추가, 완료 처리, 삭제 후 UI 반영은 즉각적으로 느껴져야 한다.
- 불필요한 새로고침 없이 주요 상호작용이 이루어져야 한다.

### 5.3 보안

- 인증은 백엔드 자체 JWT 방식으로 처리한다.
- 비밀번호는 평문으로 저장하지 않고 안전한 해시 방식으로 저장해야 한다.
- 사용자는 자신의 데이터에만 접근 가능해야 한다.
- 인증 토큰 검증 실패 시 서버는 보호된 리소스를 반환하지 않아야 한다.

### 5.4 접근성

- 키보드만으로 주요 기능 사용이 가능해야 한다.
- 입력 필드, 버튼, 체크박스는 의미 있는 레이블을 가져야 한다.
- 상태 변화와 오류 메시지는 사용자가 인지할 수 있게 제공되어야 한다.

### 5.5 안정성

- API 실패 시 사용자에게 재시도 가능한 형태의 오류 메시지를 보여야 한다.
- 잘못된 요청 데이터는 서버에서 검증되어야 한다.
- 프론트엔드는 예외 상황에서도 치명적으로 깨지지 않아야 한다.

## 10. 용어 정의

- 할 일(Task): 사용자가 등록하는 개별 작업 항목
- 완료(Completed): 사용자가 끝냈다고 표시한 상태
- 미완료(Active): 아직 끝나지 않은 상태
- 인증 토큰(JWT): 로그인 사용자 식별을 위한 토큰
