# TodoList 앱 와이어프레임

## 1. 화면 도출 요약

PRD의 사용자 스토리와 API 명세를 기준으로 MVP에 필요한 화면은 아래와 같다.

- 회원가입 화면
- 로그인 화면
- Todo 대시보드 화면
- Todo 빈 상태 화면
- 전역 오류/피드백 상태

빈 상태와 오류 상태는 별도 라우트라기보다 대시보드 및 인증 화면 안에서 나타나는 상태 화면으로 정의한다.

---

## 2. 회원가입 화면

### 근거가 되는 사용자 스토리

- `3.1 회원가입`

### ASCII 형태의 레이아웃 스케치

```text
+--------------------------------------------------+
| TodoList                                         |
| 간단하게 시작하는 개인 할 일 관리                  |
|--------------------------------------------------|
| [회원가입 카드]                                   |
|                                                  |
|  이메일                                           |
|  [ user@example.com                         ]    |
|                                                  |
|  비밀번호                                         |
|  [ ************                             ]    |
|                                                  |
|  [ 회원가입 ]                                     |
|                                                  |
|  이미 계정이 있나요? [로그인]                     |
|                                                  |
|  ! 이메일 중복 / 형식 오류 / 서버 오류 메시지       |
+--------------------------------------------------+
```

### 호출하는 API 엔드포인트

- `POST /auth/signup`

### 컴포넌트 트리

```text
SignupPage
├─ AuthShell
│  ├─ BrandHeader
│  └─ AuthCard
│     ├─ AuthTitle
│     ├─ SignupForm
│     │  ├─ EmailField
│     │  ├─ PasswordField
│     │  ├─ SubmitButton
│     │  └─ InlineErrorMessage
│     └─ AuthLinkRow
```

### 사용자 인터랙션 흐름

1. 사용자가 이메일과 비밀번호를 입력한다.
2. `회원가입` 버튼 클릭 시 클라이언트 유효성 검사를 수행한다.
3. 통과하면 `POST /auth/signup` 요청을 보낸다.
4. 성공 시 JWT를 저장하고 Todo 대시보드로 이동한다.
5. `409 EMAIL_ALREADY_EXISTS`이면 이메일 필드 아래에 중복 오류를 표시한다.
6. `422 VALIDATION_ERROR`이면 필드별 오류를 표시한다.
7. `500`이면 카드 하단에 재시도 가능한 전역 오류 메시지를 표시한다.

### 반응형 레이아웃 변화 (모바일 → 데스크톱)

- 모바일:
  단일 컬럼, 전체 폭 카드, 로고와 폼이 세로로 쌓인다.
- 태블릿:
  카드 폭이 줄고 상하 여백이 늘어난다.
- 데스크톱:
  화면 중앙 고정 카드 레이아웃, 좌우 여백을 크게 두고 브랜드 카피와 폼을 더 여유 있게 배치한다.

## 3. 로그인 화면

### 근거가 되는 사용자 스토리

- `3.2 로그인`

### ASCII 형태의 레이아웃 스케치

```text
+--------------------------------------------------+
| TodoList                                         |
| 다시 돌아온 할 일을 이어서 관리하세요               |
|--------------------------------------------------|
| [로그인 카드]                                     |
|                                                  |
|  이메일                                           |
|  [ user@example.com                         ]    |
|                                                  |
|  비밀번호                                         |
|  [ ************                             ]    |
|                                                  |
|  [ 로그인 ]                                       |
|                                                  |
|  계정이 없나요? [회원가입]                         |
|                                                  |
|  ! 인증 실패 / 형식 오류 / 서버 오류 메시지         |
+--------------------------------------------------+
```

### 호출하는 API 엔드포인트

- `POST /auth/login`

### 컴포넌트 트리

```text
LoginPage
├─ AuthShell
│  ├─ BrandHeader
│  └─ AuthCard
│     ├─ AuthTitle
│     ├─ LoginForm
│     │  ├─ EmailField
│     │  ├─ PasswordField
│     │  ├─ SubmitButton
│     │  └─ InlineErrorMessage
│     └─ AuthLinkRow
```

### 사용자 인터랙션 흐름

1. 사용자가 이메일과 비밀번호를 입력한다.
2. `로그인` 버튼 클릭 시 `POST /auth/login` 요청을 보낸다.
3. 성공 시 JWT를 저장하고 Todo 대시보드 최초 로딩을 시작한다.
4. `401 INVALID_CREDENTIALS`이면 폼 상단 또는 하단에 인증 실패 메시지를 표시한다.
5. `422 VALIDATION_ERROR`이면 필드별 오류를 표시한다.
6. 인증 토큰이 없거나 만료된 상태로 보호 페이지 접근 시 이 화면으로 리다이렉트한다.

### 반응형 레이아웃 변화 (모바일 → 데스크톱)

- 모바일:
  입력 필드와 버튼을 한 손 조작이 쉬운 큰 터치 영역으로 배치한다.
- 태블릿:
  카드 폭을 제한하고 메시지 영역을 안정적으로 확보한다.
- 데스크톱:
  회원가입 화면과 동일한 중앙 정렬 카드 패턴을 유지해 학습 비용을 낮춘다.

## 4. Todo 대시보드 화면

### 근거가 되는 사용자 스토리

- `3.3 할 일 추가`
- `3.4 할 일 목록 조회`
- `3.5 할 일 완료 처리`
- `3.6 할 일 삭제`

### ASCII 형태의 레이아웃 스케치

```text
+--------------------------------------------------------------+
| TodoList                     user@example.com   [로그아웃]    |
| 오늘 해야 할 일을 빠르게 정리하세요                            |
|--------------------------------------------------------------|
| [ 새 할 일 입력...................................... ] [추가] |
|                                                              |
| 진행 중                                                       |
| +----------------------------------------------------------+ |
| | [ ] Buy groceries                            [삭제]       | |
| |     생성: 2026-06-22 14:45                                | |
| +----------------------------------------------------------+ |
| | [ ] Review API spec                          [삭제]       | |
| |     생성: 2026-06-22 14:20                                | |
| +----------------------------------------------------------+ |
|                                                              |
| 완료됨                                                       |
| +----------------------------------------------------------+ |
| | [x] Finish PRD review                       [삭제]        | |
| |     생성: 2026-06-22 14:00                                | |
| +----------------------------------------------------------+ |
|                                                              |
| ! 저장 실패 / 불러오기 실패 / 인증 만료 메시지                |
+--------------------------------------------------------------+
```

### 호출하는 API 엔드포인트

- `GET /todos`
- `POST /todos`
- `PATCH /todos/{todoId}`
- `DELETE /todos/{todoId}`

### 컴포넌트 트리

```text
TodoDashboardPage
├─ AppShell
│  ├─ TopBar
│  │  ├─ AppLogo
│  │  ├─ UserEmailBadge
│  │  └─ LogoutButton
│  ├─ DashboardHeader
│  ├─ TodoComposer
│  │  ├─ TodoTitleField
│  │  ├─ AddTodoButton
│  │  └─ ComposerErrorMessage
│  ├─ TodoSection
│  │  ├─ SectionTitle
│  │  └─ TodoList
│  │     ├─ TodoItemCard
│  │     │  ├─ CompletionCheckbox
│  │     │  ├─ TodoTitleText
│  │     │  ├─ CreatedAtText
│  │     │  └─ DeleteButton
│  │     └─ TodoItemCard...
│  ├─ CompletedSection
│  │  ├─ SectionTitle
│  │  └─ TodoList
│  └─ GlobalFeedbackBar
```

### 사용자 인터랙션 흐름

1. 화면 진입 시 저장된 JWT로 `GET /todos`를 호출한다.
2. 성공 시 응답의 `items`를 `isCompleted` 기준으로 진행 중/완료됨 영역에 나누어 렌더링한다.
3. 사용자가 입력창에 제목을 입력하고 `추가`를 누르면 `POST /todos`를 호출한다.
4. 생성 성공 시 입력창을 비우고 새 Todo를 목록 최상단에 삽입한다.
5. 사용자가 체크박스를 클릭하면 `PATCH /todos/{todoId}`에 목표 상태를 보내고, 성공 시 해당 아이템을 다른 섹션으로 이동시킨다.
6. 사용자가 `삭제`를 누르면 `DELETE /todos/{todoId}`를 호출하고, 성공 시 카드가 즉시 목록에서 사라진다.
7. `401 UNAUTHORIZED`가 오면 토큰을 제거하고 로그인 화면으로 이동한다.
8. `404 TODO_NOT_FOUND`가 오면 해당 아이템을 목록에서 제거하고 짧은 오류 메시지를 표시한다.
9. `500`이 오면 상단 또는 하단의 전역 피드백 바에 재시도 메시지를 표시한다.

### 반응형 레이아웃 변화 (모바일 → 데스크톱)

- 모바일:
  상단 바, 입력창, 진행 중 목록, 완료 목록이 모두 단일 컬럼으로 쌓인다.
  Todo 카드 내부는 `체크박스 / 제목 / 삭제 버튼`이 세로 압축 없이 충분한 터치 간격을 유지한다.
- 태블릿:
  입력창과 추가 버튼을 한 줄에 배치하고, 콘텐츠 폭을 중앙에 제한한다.
- 데스크톱:
  대시보드 전체 폭을 넓히고 진행 중/완료됨 섹션의 여백을 넓혀 가독성을 높인다.
  필요하면 `진행 중`과 `완료됨`을 좌우 2컬럼으로 나눌 수 있지만, MVP 기본안은 상하 1컬럼 유지가 더 단순하다.

## 5. Todo 빈 상태 화면

### 근거가 되는 사용자 스토리

- `3.4 할 일 목록 조회`

### ASCII 형태의 레이아웃 스케치

```text
+--------------------------------------------------------------+
| TodoList                     user@example.com   [로그아웃]    |
|--------------------------------------------------------------|
| [ 새 할 일 입력...................................... ] [추가] |
|                                                              |
|                  아직 등록된 할 일이 없습니다                 |
|             첫 번째 Todo를 추가해서 목록을 시작하세요          |
|                                                              |
+--------------------------------------------------------------+
```

### 호출하는 API 엔드포인트

- `GET /todos`
- `POST /todos`

### 컴포넌트 트리

```text
TodoDashboardPage
├─ AppShell
│  ├─ TopBar
│  ├─ TodoComposer
│  └─ EmptyStatePanel
│     ├─ EmptyStateTitle
│     └─ EmptyStateDescription
```

### 사용자 인터랙션 흐름

1. `GET /todos` 응답의 `items`가 빈 배열이면 빈 상태 패널을 렌더링한다.
2. 사용자가 바로 입력창에 새 Todo를 추가할 수 있어야 한다.
3. `POST /todos` 성공 시 빈 상태 패널을 제거하고 일반 목록 상태로 전환한다.

### 반응형 레이아웃 변화 (모바일 → 데스크톱)

- 모바일:
  빈 상태 메시지를 입력창 아래 중앙 정렬로 배치한다.
- 데스크톱:
  여백을 더 크게 두고 빈 상태 메시지의 시각적 중심을 높인다.

## 6. 전역 오류/피드백 상태

### 근거가 되는 사용자 스토리

- `3.1 회원가입`
- `3.2 로그인`
- `3.3 할 일 추가`
- `3.4 할 일 목록 조회`
- `3.5 할 일 완료 처리`
- `3.6 할 일 삭제`

### ASCII 형태의 레이아웃 스케치

```text
+--------------------------------------------------------------+
| [!] 인증이 만료되었습니다. 다시 로그인해 주세요.      [닫기]  |
+--------------------------------------------------------------+

+--------------------------------------------------------------+
| [!] 저장에 실패했습니다. 잠시 후 다시 시도해 주세요.  [재시도] |
+--------------------------------------------------------------+

+--------------------------------------------------------------+
| [!] 입력값을 확인해 주세요.                           [닫기]  |
+--------------------------------------------------------------+
```

### 호출하는 API 엔드포인트

- `POST /auth/signup`
- `POST /auth/login`
- `GET /todos`
- `POST /todos`
- `PATCH /todos/{todoId}`
- `DELETE /todos/{todoId}`

### 컴포넌트 트리

```text
GlobalFeedbackLayer
├─ ToastMessage
├─ InlineFieldError
├─ BannerError
└─ RetryActionButton
```

### 사용자 인터랙션 흐름

1. 각 API 호출 실패 시 상태 코드별로 적절한 피드백 컴포넌트를 선택한다.
2. `422`는 폼 필드 아래 인라인 오류로 표시한다.
3. `401`은 인증 배너 표시 후 로그인 화면 이동 또는 즉시 리다이렉트한다.
4. `500`은 화면 상단 배너 또는 토스트로 표시하고 재시도 액션을 제공한다.
5. 사용자가 닫기 또는 재시도를 선택할 수 있어야 한다.

### 반응형 레이아웃 변화 (모바일 → 데스크톱)

- 모바일:
  상단 고정 배너 또는 하단 토스트 형태가 적합하다.
- 데스크톱:
  상단 배너와 인라인 오류 메시지를 함께 사용해 어느 영역에서 문제가 났는지 더 분명하게 보여준다.

## 7. 화면 간 흐름 요약

```text
[회원가입] --성공--> [Todo 대시보드]
     |                  |
     +----로그인 이동----+

[로그인] ----성공----> [Todo 대시보드]

[Todo 대시보드]
  |- 목록 없음 ------> [빈 상태]
  |- Todo 추가 ------> [일반 목록 상태]
  |- 체크박스 토글 ---> [완료/미완료 섹션 이동]
  |- 삭제 -----------> [목록에서 제거]
  |- 401 오류 -------> [로그인]
```
